# ARIA Dashboard — Demo Build Plan

**Goal:** A localhost demo that showcases LLM-powered job role analysis in real-time. User pastes a job description, watches Claude analyse and score it via BAML structured output, and sees the ARIA classification result. Scored roles accumulate into a dashboard.

## 2. BAML layer

This replaces the old multi-call scoring engine. One LLM call, structured output, streamable.

### 2.1 `baml_src/clients.baml` — Claude client

```baml

  provider "anthropic"
  options {
    model "claude-sonnet-4-20250514"
    api_key env.ANTHROPIC_API_KEY
  }
}
```

### 2.2 `baml_src/score_role.baml` — Types + function

```baml
class VariableScore {
  score int @description("Score from 0-100")
  justification string @description("2-3 sentences citing specific tasks from the role")
}

class AISScores {
  cognitive_routine_level VariableScore @description("Degree to which cognitive tasks follow predictable, rule-based patterns. High = highly routine.")
  data_dependency VariableScore @description("Extent to which the role works with structured, machine-readable data. High = heavily data-dependent.")
  process_repeatability VariableScore @description("Degree to which processes are standardised, repeatable, high-volume. High = highly standardised.")
  social_perception VariableScore @description("Degree to which role requires reading social cues, empathy, negotiation. High = high social requirement.")
  physical_complexity VariableScore @description("Degree to which role requires physical presence, dexterity. High = high physical requirement.")
  regulatory_accountability VariableScore @description("Degree to which role carries personal regulatory/legal/fiduciary accountability. High = high accountability.")
}

class APSScores {
  knowledge_processing VariableScore @description("Volume and complexity of information the role must synthesise or produce. High = heavy knowledge work.")
  output_sensitivity VariableScore @description("Degree to which outputs benefit from higher volume or improved quality. High = significant gains from AI.")
  decision_support VariableScore @description("Extent to which role makes/supports complex decisions benefiting from data analysis. High = heavy decision needs.")
  repetitive_cognitive VariableScore @description("Amount of repetitive analytical/cognitive tasks within an expert-level role. High = lots of repetitive expert work.")
  communication_volume VariableScore @description("Volume of communication, coordination, collaboration the role manages. High = heavy communication load.")
}

class RoleAnalysis {
  tasks string[] @description("5-15 discrete tasks extracted from the role description")
  ais AISScores
  aps APSScores
}

function ScoreRole(title: string, department: string, description: string) -> RoleAnalysis {
  client Claude
  prompt #"
    You are an AI workforce impact analyst. You assess job roles for automation potential
    and AI augmentation potential using a structured methodology.

    ## Role
    **Title:** {{ title }}
    **Department:** {{ department }}
    **Description:** {{ description }}

    ## Instructions
    1. First, extract 5-15 discrete tasks from the role description.
    2. Then score all Automation Impact Score (AIS) variables (0-100 each):
       - Cognitive routine level: How much cognitive work follows predictable, rule-based patterns
       - Data dependency: How much the role works with structured, machine-readable data
       - Process repeatability: How standardised and high-volume the processes are
       - Social perception: How much the role requires empathy, negotiation, interpersonal nuance
       - Physical complexity: How much physical presence, dexterity is required
       - Regulatory accountability: How much personal regulatory/legal accountability the role carries
    3. Then score all Augmentation Potential Score (APS) variables (0-100 each):
       - Knowledge processing: Volume/complexity of information to synthesise
       - Output sensitivity: How much outputs benefit from higher quality/volume
       - Decision support: Extent of complex decisions benefiting from data analysis
       - Repetitive cognitive: Amount of repetitive analytical work within expert role
       - Communication volume: Volume of coordination and collaboration

    Scoring rubric for each variable:
    - 0-20:  Minimal — barely present in the role
    - 21-40: Low — occasional instances, not defining
    - 41-60: Moderate — regularly present but balanced by other factors
    - 61-80: High — dominant characteristic, significantly defines the role
    - 81-100: Very high — the primary defining feature of the role

    Cite specific tasks in each justification.

    {{ ctx.output_format }}
  "#
}
```

### 2.3 How BAML integrates with the backend

```python
# backend/main.py (scoring endpoint)
from baml_client.async_client import b
from baml_client.types import RoleAnalysis

async def score_role_stream(title, department, description):
    stream = b.stream.ScoreRole(title, department, description)
    async for partial in stream:
        # partial is a PartialRoleAnalysis — fields populate as LLM streams
        # partial.tasks may have some items, partial.ais.cognitive_routine_level may exist
        yield partial
    final = await stream.get_final_response()  # complete RoleAnalysis
    return final
```

**Key advantage:** BAML streaming gives us partial results as the LLM generates them. The frontend sees tasks appear first, then AIS scores fill in one by one, then APS scores — all from a single LLM call. BAML's `PartialRoleAnalysis` type has all fields as Optional, so we can detect which fields are newly populated and push updates to the frontend.


## 3. Backend

### 3.1 Database (SQLite + SQLAlchemy)

Three tables, simplified from the full spec:

```sql
roles
  id              INTEGER PRIMARY KEY
  title           TEXT NOT NULL
  department      TEXT
  grade           TEXT
  headcount       INTEGER DEFAULT 1
  description     TEXT NOT NULL
  tasks           TEXT          -- JSON array of extracted tasks
  ais_composite   REAL
  aps_composite   REAL
  classification  TEXT          -- e.g. "Transform", "Maintain"
  ais_band        TEXT          -- low / medium / high
  aps_band        TEXT          -- low / medium / high
  risk_level      TEXT          -- Very high / High / Moderate / Low / Very low
  created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

ais_variables
  id          INTEGER PRIMARY KEY
  role_id     INTEGER REFERENCES roles(id)
  variable    TEXT NOT NULL     -- crl / dds / prv / sp / pc / ra
  name        TEXT NOT NULL     -- human-readable name
  raw_score   INTEGER
  is_inverse  BOOLEAN
  adjusted    INTEGER           -- after inverse (if applicable)
  weight      REAL
  weighted    REAL
  rationale   TEXT

aps_variables
  id          INTEGER PRIMARY KEY
  role_id     INTEGER REFERENCES roles(id)
  variable    TEXT NOT NULL     -- kip / oqv / dsr / rcw / ccv
  name        TEXT NOT NULL
  score       INTEGER
  weight      REAL
  weighted    REAL
  rationale   TEXT
```

File: `backend/database.py`
- SQLAlchemy models for the three tables
- `init_db()` function that creates tables on startup
- SQLite file stored at `backend/aria.db`

### 3.2 Classifier (pure Python)

File: `backend/classifier.py`

Takes the raw `RoleAnalysis` from BAML and computes:

1. **AIS composite** — apply inverse for social_perception, physical_complexity, regulatory_accountability, then weighted sum:
   ```
   AIS = crl×0.25 + dds×0.20 + prv×0.20 + (100-sp)×0.15 + (100-pc)×0.10 + (100-ra)×0.10
   ```

2. **APS composite** — direct weighted sum:
   ```
   APS = kip×0.25 + oqv×0.25 + dsr×0.20 + rcw×0.20 + ccv×0.10
   ```

3. **Band assignment** — Low (0–44), Medium (45–64), High (65–100)

4. **Matrix classification** — lookup from (ais_band, aps_band) → one of 9 classifications

5. **Risk level** — Very high (≥75), High (≥60), Moderate (≥45), Low (≥30), Very low (<30)

### 3.3 API endpoints

File: `backend/main.py`

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/score-role` | Accept `{title, department, description}`. Stream BAML partial results via SSE, then classify + persist. |
| GET | `/api/roles` | Return all scored roles with composite scores and classifications. Optional `?dept=` filter. |
| GET | `/api/roles/{id}` | Return single role with full variable breakdowns. |
| GET | `/api/summary` | Return `{total_roles, by_classification: {}, by_risk: {}, matrix_counts: {}}`. |
| DELETE | `/api/roles/{id}` | Delete a role (convenience for demo). |

### 3.4 SSE stream format for `/api/score-role`

The endpoint streams partial BAML results as they arrive from the LLM. The backend compares each partial to the previous one and emits events for newly populated fields.

```jsonl
event: status
data: {"message": "Analysing role..."}

event: tasks
data: {"tasks": ["Task 1", "Task 2", "Task 3", ...]}

event: ais_variable
data: {"variable": "cognitive_routine_level", "name": "Cognitive routine level", "score": 78, "justification": "..."}

event: ais_variable
data: {"variable": "data_dependency", "name": "Data dependency", "score": 72, "justification": "..."}

... (remaining AIS variables as they stream in)

event: aps_variable
data: {"variable": "knowledge_processing", "name": "Knowledge processing", "score": 82, "justification": "..."}

... (remaining APS variables as they stream in)

event: complete
data: {
  "role_id": 1,
  "ais_composite": 74.6,
  "aps_composite": 73.9,
  "classification": "Transform",
  "ais_band": "high",
  "aps_band": "high",
  "risk_level": "Very high"
}
```

**Implementation detail:** In the SSE generator function, iterate `async for partial in stream`, diff against previous partial, and yield SSE events for any newly populated fields. After the stream completes, run the classifier, persist to SQLite, and emit the `complete` event.

### 4.3 Components

**ScoreRoleForm.tsx** — Modal/dialog triggered by "+ Score a Role" button
- Fields: Role title (text), Department (text), Description (textarea, large)
- Optional: Grade, Headcount
- Submit button → calls POST `/api/score-role` and switches to ScoringProgress view

**ScoringProgress.tsx** — Shown after form submission, consumes SSE stream
- Displays extracted tasks as a list when the `tasks` event arrives
- Shows each variable score appearing in real-time as `ais_variable` / `aps_variable` events arrive:
  - Variable name on the left, score bar filling in, justification fading in
  - AIS section (6 rows), then APS section (5 rows)
  - Rows start greyed out, light up when their score arrives
- On `complete` event: classification badge animates in with composite scores and recommendation text
- "View in Dashboard" button to close and refresh the dashboard

**AriaMatrix.tsx** — 3×3 grid
- Rows: APS High / Medium / Low (top to bottom)
- Columns: AIS Low / Medium / High (left to right)
- Each cell: classification name, role count, coloured background
  - Red shades: Transform, Accelerate, Transition (high priority)
  - Amber shades: Optimize, Adapt, Monitor (medium)
  - Green shades: Expand, Invest selectively, Maintain (low)
- Clicking a cell filters the role table below

**RoleTable.tsx** — Table of all scored roles
- Columns: Title, Department, AIS (with colour), APS (with colour), Classification (badge), Risk Level (badge)
- Sortable by clicking column headers
- Click row → open RoleDetail panel
- Filterable by matrix cell click or department

**RoleDetail.tsx** — Slide-in panel from right
- Header: title, department, classification badge
- Two score gauges: AIS and APS (simple coloured bars or semicircle gauges)
- AIS variable breakdown: 6 rows, each with variable name, raw score, inverse indicator, weight, weighted score, horizontal bar, rationale text
- APS variable breakdown: 5 rows, same format minus inverse
- Strategic recommendation text block (from classification definitions in spec section 4.3)
- Delete button (for demo cleanup)

**ScatterPlot.tsx** — Recharts scatter chart
- X-axis: AIS (0–100), Y-axis: APS (0–100)
- Reference lines at 44 and 65 on both axes (band boundaries), creating the 9-zone grid
- Each dot = one role, coloured by classification
- Tooltip on hover: role title, AIS, APS, classification
- Dot size scaled by headcount (optional, can default to uniform)

**MetricCards.tsx** — 4 summary cards
- Total roles assessed
- Roles at high automation risk (AIS ≥ 65)
- Roles with high augmentation potential (APS ≥ 65)
- Priority action roles (Transform + Accelerate + Transition count)

## 6. Running the demo

```bash
# One-time setup
cd backend && pip install -r requirements.txt
baml-cli generate   # generates baml_client/ from baml_src/
cd ../frontend && npm install

# Terminal 1: Backend
cd backend
export ANTHROPIC_API_KEY=sk-ant-...
uvicorn main:app --reload --port 8000

# Terminal 2: Frontend
cd frontend
npm run dev
# → http://localhost:5173
```
